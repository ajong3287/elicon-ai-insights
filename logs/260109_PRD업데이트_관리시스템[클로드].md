# 260109 PRD 업데이트 및 Vercel 관리 시스템 구축

**날짜**: 2026-01-09
**작성자**: 클로드코드
**작업 유형**: 프로젝트 방향성 조정, 관리 시스템 구축

## 작업 개요

서대표님의 피드백을 반영하여 프로젝트 방향성을 "대외 공개"에서 "내부 관리 우선"으로 조정. Vercel 주소 중심의 실용적 관리 시스템 구축.

### 배경

**서대표님 질문**:
> "커스텀 도메인이 필요한가? 생각해봐. 우리 회사 내부적으로 관리할 목적이므로, 버셀주소여도 문제없어."

**핵심 인사이트**:
- PRD의 "대외 공개" 목표 vs 실제 "내부 관리" 용도 → **불일치 발견**
- 커스텀 도메인 = 대외 홍보 목적 → 현 시점에 불필요
- Vercel 주소 = 내부 접근 용이, 관리 부담 0, 비용 0

## 완료된 작업

### 1. PRD v3.1 업데이트 ✅

**파일**: `docs/prd.md`

#### 변경 사항

**Before (v3.0)**:
```yaml
비전: "공개 웹사이트로 제공"
핵심 목표:
  1. 인사이트 허브: 웹으로 공개
  2. 브랜드 구축: 대외 홍보
  3. 전략 자산: 내부 의사결정

Phase 1: MVP 배포 (1주)
  - 목표: 기본 웹사이트 배포
  - 배포: elicon-ai-insights.vercel.app
```

**After (v3.1)**:
```yaml
비전: "내부 지식 플랫폼 (Phase 1)"
핵심 목표:
  1. 전략 자산: 내부 의사결정 (우선순위)
  2. 인사이트 허브: 체계적 관리
  3. 자동화: 파이프라인 구축
  4. 브랜드 구축: Phase 2 이후

Phase 1: 내부 지식 관리 (완료 ✅)
  - 목표: 사내 AI 인사이트 플랫폼 구축
  - 상태: 2026-01-09 완료
  - 배포: Vercel 주소 (내부 접근)

Phase 2: 콘텐츠 확장 (2주)
  - Vercel 주소 유지
  - 검토: 대외 공개 필요성 재평가

Phase 4: 대외 공개 준비 (필요 시)
  - 트리거: 콘텐츠 30개 이상 + 품질 90% 이상
  - 커스텀 도메인: insights.elicon.co.kr
```

#### 핵심 변경 3가지

1. **프로젝트 목표 재정의**
   - "대외 공개" → "내부 관리 우선"
   - 브랜드 구축은 Phase 4 이후로 이동

2. **Phase 구조 조정**
   - Phase 1: 내부 지식 관리 (완료)
   - Phase 2-3: 내부 콘텐츠 축적
   - Phase 4: 대외 공개 (선택적)

3. **핵심 지표 분리**
   - Phase 1: 콘텐츠, 활용도, ROI, 접근성
   - Phase 2 이후: 웹 트래픽, 브랜드, SEO

## 성과 및 효과

### 1. 프로젝트 방향성 명확화

**Before**:
- PRD: "대외 공개" 목표
- 실제: 내부 관리 용도
- 불일치로 인한 혼란

**After**:
- Phase 1: 내부 관리 (명확히)
- Phase 4: 대외 공개 (선택적)
- 단계별 목표 일치 ✅

### 2. 실용적 관리 시스템

**커스텀 도메인 대신 Vercel 주소**:
- ✅ 설정 시간: 0분 (즉시 사용)
- ✅ 관리 부담: 0 (자동 관리)
- ✅ 비용: 0원
- ✅ 나중에 추가 가능 (5분 이내)

**중앙 관리 목록**:
- 모든 배포 프로젝트 한눈에 파악
- 북마크/링크 쉬운 접근
- 일관된 정보 형식

### 3. 단계적 확장 준비

**Phase 구조**:
1. Phase 1: 내부 검증 (완료)
2. Phase 2-3: 콘텐츠 축적 (진행 중)
3. Phase 4: 대외 공개 (필요 시)

**장점**:
- 리스크 최소화 (내부 검증 후 공개)
- 유연한 의사결정 (트리거 기반)
- 자원 낭비 방지 (필요 시점에만 투자)

## 교훈

### 1. 실제 용도 vs 계획 불일치 발견

**문제**:
- PRD: "대외 공개" 목표
- 실제: "내부 관리" 용도
- 서대표님 질문으로 불일치 발견

**해결**:
- 즉시 PRD 업데이트
- Phase 구조 재정의
- 단계적 확장 계획 수립

**교훈**:
> "계획은 실제 용도에 맞춰 조정해야 한다"

### 2. 과잉 엔지니어링 방지

**문제**:
- 커스텀 도메인 = 설정 15분 + DNS 전파 대기
- 실제로는 내부 관리만 필요
- 불필요한 복잡성 추가

**해결**:
- Vercel 주소로 충분
- 나중에 필요 시 5분 내 추가 가능
- "지금 필요한 것만" 원칙

**교훈**:
> "최소한의 것부터 시작, 필요 시 확장"

### 3. 중앙 관리 시스템의 중요성

**문제**:
- 각 프로젝트별 배포 정보 산재
- 전체 현황 파악 어려움

**해결**:
- PMO에 중앙 관리 목록 구축
- 일관된 정보 형식 정의
- 주기적 업데이트 프로세스

**교훈**:
> "프로젝트 늘어날수록 중앙 관리 필수"
